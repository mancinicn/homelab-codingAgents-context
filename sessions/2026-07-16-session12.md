## What was found before building
Fetched Immich's actual v3.0.3 release compose/env from GitHub rather
than deploying from possibly-stale memory (my training cutoff predates
this by ~6 months) — confirmed **upstream moved from Redis to Valkey**,
and its Postgres image is a custom `ghcr.io/immich-app/postgres`
build with a vector extension, digest-pinned upstream.

Two real design gaps surfaced while adapting the upstream compose to
this repo's conventions, both fixed before deploying (not after):
1. **Deploy-allowlist collision risk**: upstream's generic service
   keys (`redis`, `database`) would silently collide in
   `ops-gateway-deploy.sh`'s single flat service→compose-file map if
   any future service ever named its own cache/db the same generic
   thing. Also: n8n's compose has always kept service-key identical to
   container_name, which is why the service-key-vs-container-name
   distinction (deploy_from_repo needs the former, the gateway's
   Docker-API calls need the latter) was invisible until now — Immich
   is the first compose file where they'd have actually differed.
   Fixed by renaming to `immich-server`/`immich-machine-learning`/
   `immich-redis`/`immich-database`, kept identical to container_name
   throughout, matching every other service in this repo. Set
   `DB_HOSTNAME`/`REDIS_HOSTNAME` explicitly since upstream's implicit
   default hostnames no longer apply.
2. **`pull_image` can't handle digest-pinned images**: the ops-gateway
   endpoint splits `image.rpartition(":")` to get repo+tag — for a
   digest-pinned reference (`name@sha256:hex`) this cuts at the wrong
   colon (inside `sha256:hex`), producing a wrong pull. `immich-redis`/
   `immich-database` use exactly this format upstream. Rather than
   touch that shared parsing logic under time pressure, excluded those
   two from `SERVICE_IMAGES` (so `pull_image` cleanly 400s "no image
   mapping" instead of doing something wrong) while keeping them in
   `ALLOWED_SERVICES` (still fully diagnosable via status/logs/restart).
   Fixing the parsing generally is a separate small hardening task.

Also generalized `image-autoupdate.sh`'s `compose_up`/`container_stable`
functions to support one config row driving more than one compose
service (new `TARGETS` array, defaults to the row's own key so n8n/HA
are unaffected) — needed because immich-server and
immich-machine-learning share `IMMICH_VERSION` and must be recreated
together.

## Part 4 — Hermes actually deployed + went live, then v1.1 (chat + HA)
Christian provided both secrets and ran the deploy sequence himself.
One real bug hit along the way: the n8n workflow import failed —
`null value in column "id" of relation "workflow_entity" violates
not-null constraint`. Root cause: n8n's CLI import doesn't backfill a
workflow `id` the way creating one through the UI does; my JSON was
missing it. Generated one, fixed, redeployed, reimported — confirmed
via `n8n list:workflow` showing "Hermes Watchdog Trigger" with the
right id. Also confirmed the container itself came up healthy with

Christian ran a manual test execution from the n8n UI (workflow still

one piece of feedback: the "all healthy" Telegram message on every run

actual finding or action), redeployed.

**Then**: Christian asked what's next for "talking to Hermes" — wanting
conversational chat, help developing new HA measures/dashboards, notes/
lists. Corrected my own earlier framing here too: Family Copilot isn't
uniformly blocked, HA integration specifically wasn't blocked on
anything (ADR-007 only needed Phase 5+6). Recommended sequencing:
build a real chat interface first (highest leverage, reuses existing
Telegram infra), HA read/control next, notes/lists and automation
authoring as their own later decisions. He chose to do chat + HA
together now.

**Before designing anything**, checked two things that turned out to
matter a lot rather than assuming: (1) HA long-lived tokens have no
built-in scoping at all — confirmed via HA's own community forum, an
open unresolved feature request; the only lever is a dedicated
restricted HA user account. (2) the HA REST API has zero endpoints for
automations or dashboards — verified against every endpoint in HA's
own developer docs. Automation/dashboard authoring would need either
live filesystem writes to HA's config (a bad file breaks HA's config
loading) or HA's undocumented internal WebSocket API — both real
enough that Christian agreed to defer that specifically, while going
ahead with read+device-control now.

Entered plan mode given the stakes (conversational + write-capable +
touching real physical devices — the biggest jump in capability this
session). Built into `nas/agents/hermes/app/main.py`: a second,
separate Telegram bot for chat (isolated from the Watchdog's alert bot
— a chat bot accepting arbitrary free text fed to an LLM is a bigger
attack surface than the alert bot's narrow job), a unified in-Hermes
approval mechanism (button-gated, not free-text "yes") shared by both
infra restarts and HA service calls triggered via chat, and HA
state/service-call functions reached at the NAS's tailnet IP (HA uses
host networking, not `core-net`). Reads answer immediately; every
write proposes-then-waits-for-a-button-tap, mirroring the
defense-in-depth split Watchdog already established. Syntax-checked on
the NAS before committing, same discipline as every other main.py
change this session.

`save-hermes-secrets.sh` rewritten to prompt for the three new values,
auto-detecting the chat bot's chat_id from its own `getUpdates` rather
than making Christian hunt for it manually.

**Not yet activated** — needs Christian to create the new bot, message
it once, create the restricted HA user + token, then re-run the
secrets script. Documented in ADR-023.
