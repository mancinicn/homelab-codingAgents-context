## What was found before building
Fetched Immich's actual v3.0.3 release compose/env from GitHub rather
than deploying from possibly-stale memory (my training cutoff predates
this by ~6 months) — confirmed **upstream moved from Redis to Valkey**,
and its Postgres image is a custom `ghcr.io/immich-app/postgres`
build with a vector extension, digest-pinned upstream.

Two real design gaps surfaced while adapting the upstream compose to
this repo's conventions, both fixed before deploying (not after):
1. **Deploy-allowlist collision risk**: upstream's generic service
   keys (`redis`, `database`) would silently collide in
   `ops-gateway-deploy.sh`'s single flat service→compose-file map if
   any future service ever named its own cache/db the same generic
   thing. Also: n8n's compose has always kept service-key identical to
   container_name, which is why the service-key-vs-container-name
   distinction (deploy_from_repo needs the former, the gateway's
   Docker-API calls need the latter) was invisible until now — Immich
   is the first compose file where they'd have actually differed.
   Fixed by renaming to `immich-server`/`immich-machine-learning`/
   `immich-redis`/`immich-database`, kept identical to container_name
   throughout, matching every other service in this repo. Set
   `DB_HOSTNAME`/`REDIS_HOSTNAME` explicitly since upstream's implicit
   default hostnames no longer apply.
2. **`pull_image` can't handle digest-pinned images**: the ops-gateway
   endpoint splits `image.rpartition(":")` to get repo+tag — for a
   digest-pinned reference (`name@sha256:hex`) this cuts at the wrong
   colon (inside `sha256:hex`), producing a wrong pull. `immich-redis`/
   `immich-database` use exactly this format upstream. Rather than
   touch that shared parsing logic under time pressure, excluded those
   two from `SERVICE_IMAGES` (so `pull_image` cleanly 400s "no image
   mapping" instead of doing something wrong) while keeping them in
   `ALLOWED_SERVICES` (still fully diagnosable via status/logs/restart).
   Fixing the parsing generally is a separate small hardening task.

Also generalized `image-autoupdate.sh`'s `compose_up`/`container_stable`
functions to support one config row driving more than one compose
service (new `TARGETS` array, defaults to the row's own key so n8n/HA
are unaffected) — needed because immich-server and
immich-machine-learning share `IMMICH_VERSION` and must be recreated
together.

## Build + deploy
New `nas/media/immich/` (first "media" category — existing ones are
`automation/`, `home/`), `nas/scripts/save-immich-secrets.sh`
(generates one random password, writes it under both `DB_PASSWORD` and
`POSTGRES_PASSWORD` keys — deliberately NOT via compose-level
`${DB_PASSWORD}` interpolation, which would hit the exact same
git-tracked-`.env`-vs-secret-`env_file` gotcha found in ADR-015 for
Traefik's Cloudflare token the day before). Registered with
ops-gateway, the auto-updater, and appdata backup (pg_dump + library
path added, model-cache excluded as re-downloadable).

Deployed: scp'd everything to the NAS, Christian ran
`save-immich-secrets.sh`, `docker compose up -d` for Immich (all 4
containers pulled/started clean first try), then
`docker compose up -d --build` for ops-gateway to pick up the new
`main.py` — syntax-checked with `python3 -m py_compile` on the NAS
before asking him to rebuild it, since it's the container that gives
this session its own NAS access.

**Verified end-to-end**: all 4 Immich containers + ops-gateway healthy;
`/api/server/ping` returns 200; ops-gateway's own
`service_status/immich-server` and `.../immich-machine-learning` return
correct live data through the real bearer-token path (not just
container inspection) — confirms the whole registration actually
works, not just that the config looks right; a bad service name
correctly 400s. RAM settled to ~3GB available (from ~4.2GB baseline)
with everything running — within the 2026-07-14 reassessment's
validated envelope.
