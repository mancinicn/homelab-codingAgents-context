## Workstream B — deploy_from_repo (ADR-018)
Resolved the twice-punted Phase 7 remainder with a THIRD option that
beats both ADR-012 choices: the Phase 8 reboot file-trigger pattern.
- `POST /deploy/{name}?approval_id=…` on the ops-gateway (v1.5):
  allowlist-validates the name, requires a Telegram approval, writes
  only the name to a trigger file. A root host systemd unit
  (ops-gateway-deploy.path/.service → ops-gateway-deploy.sh) reads it,
  re-validates against its own identical allowlist, runs

  documented v2 (needs a git credential on the NAS).

## RAM reassessment + reprioritization (2026-07-14)
Christian asked to reassess roadmap priorities against RAM and whether
anything could move NAS→VPS to make room for Immich.
- **Immich is pinned to the NAS** by data locality (photos on the 4TB
  NAS array; VPS is the 96GB edge box). Can't move Immich to VPS, and
  nothing else usefully moves NAS→VPS: n8n is canonically NAS (ADR-003),
  HA needs LAN/host-net, ops-gateway/outpost are NAS-local. Moving
  compute to the edge box also fights the "VPS = minimal edge/identity"
  model. So the "move to VPS" lever doesn't apply here.
- NAS RAM at reassessment: 7.5GB total, ~4.3GB available. VPS: 7.8GB,
  ~5.9GB available (lots of headroom; legacy n8n-zuij wastes ~319MB).
- **Immich ML RAM myth corrected**: ML is NOT a constant 2–4GB — the
  model unloads after 5 min idle (MACHINE_LEARNING_MODEL_TTL=300 default);
  the 2–4GB is a processing PEAK. Heavy phase is the one-time initial
  import (run overnight). So ML-ON is likely feasible on 8GB; a 16GB
  upgrade (DXP2800 takes user-replaceable DDR5-4800 SODIMM to 16–32GB,
  ~€40–80) is optional headroom, not a blocker.
- **Reprioritized**: auto-update controller is NEXT (RAM-cheap, builds
  on deploy_from_repo, no memory competition), Immich (Phase 9) after,
  with the import/access/backup decisions to settle in its own plan.
  Opportunistic: retire n8n-zuij (VPS hygiene). Roadmap updated.
