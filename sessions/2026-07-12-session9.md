## Real findings from actual testing (not assumed)
1. **docker-socket-proxy has no separate DELETE toggle** ŌĆö `POST=1`
   (on since Phase 7) already permitted container create/remove via
   `CONTAINERS=1`, confirmed by `remove_container` working with no
   proxy change beyond adding `VOLUMES=1`. Not a new grant, just newly
   documented (mirrors ADR-012's IMAGES+POST finding).
2. **`volumes/prune` only removes anonymous volumes by default** ŌĆö
   first prune test left a real throwaway named volume untouched.
   Fixed with `filters={"all":"true"}`, re-verified with a second real
   test that correctly swept it.
3. **This NAS's Docker state doesn't reliably survive a host reboot**
   ŌĆö the big one. The real reboot test (genuine NAS reboot, approved
   via Telegram) came back with two unrelated pieces of corrupted
   per-container state:
   - `n8n-outpost-redis`'s anonymous `/data` volume owned by
     `dnsmasq:1000` (unrelated system user) at mode `0000` ŌĆö
     unreadable by anyone. Crash-looped (`restart_count` 93 within
     ~90 min) trying to read `dump.rdb` on every restart.
   - `n8n-outpost`'s Docker-generated `resolv.conf` came back
     unreadable inside the container, causing it to fall back to a
     bogus `::1:53` resolver and never reach Authentik ŌĆö the family
     n8n gate was silently non-functional for ~1h45m (container stayed
     "running" throughout, so nothing alerted on it)

   Diagnosed by elimination: a fresh throwaway container on the same
   network resolved DNS correctly the whole time, ruling out a
   Docker-daemon-wide fault. Both fixed by recreating (not restarting
   ŌĆö restart reuses existing broken files, same reason `docker
   restart` doesn't re-read `env_file` either) the affected
   containers: `docker rm -f` + `docker volume rm` + `compose up -d`
   for redis (pure session cache, safe to lose), `compose up -d
   --force-recreate` for the outpost (config unchanged, so plain
   `up -d` would have skipped it).

   Root cause NOT identified this session ŌĆö candidates include
   btrfs/mount timing on `/volume1` during boot. Documented as an open
   loose end in ADR-014 and HANDOVER.md. Real operational risk: don't
   rely on `reboot` unattended (Hermes, Phase 10) until this is
   understood.
4. **Separate, VPS-side finding, unrelated to the above**: once the
   NAS-side fixes let the outpost actually reach the VPS, its config-
   fetch API (`/api/v3/outposts/instances/`) returned a consistent 504
   Gateway Timeout, even though `authentik-server`'s own container
   healthcheck is green. All VPS containers healthy/up for days ŌĆö not
   a crash. Likely pre-existing and simply invisible while the outpost
   couldn't reach the VPS at all. Christian chose to park this for
   separate investigation rather than continue debugging under Phase
   8's scope ŌĆö family n8n gate (5679) stays degraded, Christian's own
   direct access (5678) unaffected.

## Part 2: Authentik/Traefik 504 investigation (same session, resolved)
Christian asked to come back to the parked Authentik issue after the
Phase 8 docs were committed. Full root-cause chase, documented in
ADR-015:

- Ruled out, in order, with real tests at each step: DNS, network path
  (localhost-on-VPS failed identically, so never a path issue), TCP
  handshake (fine), TLS certificate (valid, handshake succeeds via raw
  openssl), authentik-server health (fully healthy ŌĆö its own embedded
  outpost was calling it successfully every 5 min over loopback the
  whole time), traefikŌåÆauthentik-server connectivity via `wget` from
  inside the traefik container (worked fine, by name and by IP),
  duplicate/conflicting router definitions (none), a plain
  `docker restart traefik` (didn't fix it), a full `--force-recreate`
  of traefik (didn't fix it either)
- Conclusive proof it was Traefik-internal: tailed authentik-server's
  logs live while firing a request through Traefik ŌĆö the request never
  arrived at the backend at all, only the container's own internal
  healthcheck showed up
- Enabled Traefik debug logging temporarily (`--log.level=DEBUG`) ŌĆö
  immediately showed the real cause: `dial tcp 172.20.0.4:9000: i/o
  timeout`. authentik-server is on two Docker networks (`proxy` +
  `authentik-internal`); Traefik's Docker provider was picking the
  network Traefik itself isn't even attached to. Fixed with
  `traefik.docker.network=proxy` on authentik-server. Verified: hang ŌåÆ
  `HTTP 302` in ~170ms
- Along the way, discovered and fixed a second, unrelated latent bug:
  Traefik's ACME email was being set to a blank string on every
  recreate (a missing `.env` file for Compose's own `${...}`
  interpolation in `command:` args ŌĆö a different mechanism from the
  container's `env_file`, which was fine). Created
  `/opt/vps-infra/edge/.env` with just `LETSENCRYPT_EMAIL`
- Reverted debug logging back to `INFO` after diagnosis
- New finding: `/opt/vps-infra/` (edge/traefik.yml,
  identity/authentik.yml, tools/vaultwarden.yml) isn't tracked in this
  git repo at all ŌĆö changes had to be made directly on the VPS with no
  diff/rollback path. Worth importing as a follow-up

## Part 5: reboot-survival investigation ŌĆö real incident found + likely root cause
Christian chose to pursue the reboot-survival follow-up next. Before
even getting into the boot-order investigation, spotted something
critical in a routine log check: `n8n` itself had been silently
crash-looping for 4.5+ hours (`getaddrinfo EAI_AGAIN n8n-postgres`,
restart_count past 970) ŌĆö the same DNS-resolution symptom as this
morning's `n8n-outpost` fix, but on a completely different network,
with massively delayed onset, and undetected the whole time
(Christian's own direct n8n access was down, not just the family
gate). Fixed the same way (`docker compose up -d --force-recreate
n8n`), verified clean. Given the delayed-onset risk, also proactively
recreated `n8n-postgres` and `homeassistant` ŌĆö neither had been
recreated since the reboot (`started_at` still matched 14:12), so
both were carrying unknown risk even with zero visible symptoms so
far. Both came back clean; full sweep confirmed all 5 NAS containers
now on fresh post-reboot state.

Then continued into the actual boot-sequence investigation:
- `docker.service` has no ordering dependency on `/volume1` at all
- `/volume1` isn't even a normal static systemd mount (not in
  /etc/fstab, discovered only at runtime via a UGOS-proprietary
  LVM/device-mapper volume) ŌĆö no reliable ordering guarantee exists
  at the systemd level between storage readiness and Docker startup
- The more direct, evidenced culprit: UGOS's own `index_serv`
  (NAS web UI file-indexing daemon) watches every write across
  `/volume1`, including deep inside Docker's `overlay2` storage. Its
  logs show a genuine bug in its own exclusion logic ŌĆö
  `sharefolder.IsShareFolder(/volume1/appdata) got 0 with error:
  strconv.ParseUint: parsing "": invalid syntax` ŌĆö because
  /volume1/docker and /volume1/appdata were hand-configured (ADR-008)
  rather than created as real UGOS Shared Folders, so index_serv can't
  look up their type and its "should I skip this" check is undefined.
  **748 occurrences of this exact error in 6 hours** ŌĆö confirmed
  continuously firing, not a one-off boot fluke, meaning the
  underlying race isn't strictly reboot-specific (a reboot's burst of
  simultaneous writes just makes hitting it far more likely)
- Suggested fix, not yet applied: register both paths as real UGOS
  Shared Folders via the web UI (needs Christian, no agent access to
  that UI), then re-verify with another real reboot test

Full writeup in ADR-014 (appended, not a new ADR ŌĆö same investigation
thread as this morning's findings).
