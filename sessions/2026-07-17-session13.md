## The bot-token troubleshooting saga
Christian's first several attempts to save the chat bot token all
failed silently — `save-hermes-secrets.sh` reported success, but the
chat interface never responded and no greeting arrived. Root-caused in
stages, each one a real finding:

1. **A genuine script bug**: the original `save-hermes-secrets.sh`
   crashed with "unbound variable" at the very end (`unset` ran before
   a line that referenced the just-unset var) — fixed, but not the
   actual cause of the silent failure.
2. **The real cause**: this terminal corrupts long strings pasted
   under `read -s` (hidden input) specifically — proved by having
   Christian paste the identical token into a browser URL bar (worked,
   `"ok":true`) vs. a visible (non-`-s`) terminal `read` (also worked)
   vs. the hidden prompt (consistently failed, `404`). Fixed by making
   the script validate against Telegram's `getMe` and automatically
   retry with a visible prompt on failure, rather than a one-off manual
   workaround.
3. Along the way, added: idempotent re-runs (blank = keep existing
   value, so adding the HA token later won't require re-pasting
   everything), and a direct manual chat_id entry option (auto-detection
   via `getUpdates` proved flaky — the chat_id is just a number, not
   secret, safe to type directly; turned out to be Christian's own
   Telegram user id, same across every bot, which he already knew).
4. **Telegram's own send restriction**: a bot can't message a user who
   hasn't started that specific conversation first — hit this exactly
   once, unsurprising in hindsight, resolved by messaging the bot
   before restarting Hermes.
5. Christian tested with a spare pre-existing bot ("OpenClawMancibot",
   from an unrelated earlier project) partway through to isolate
   whether the problem was bot-specific — good instinct, confirmed it
   wasn't.
6. **The bots ended up backwards relative to their Telegram names** —
   the bot literally named "Hermes" was serving the shared `notify.env`
   channel (Watchdog alerts, ops-gateway approvals), while the bot
   named "Claude" was serving Hermes's own chat. Asked to swap them.
7. Wrote `swap-hermes-notify-bots.sh` (backs up both files, swaps just
   the Telegram fields). Ran clean — but the "swap" turned out to be a
   no-op: both files had already converged on the same bot token
   (`Hermes_mancibot`) by the time it ran, confirmed by checking both
   files' actual bot identity side by side rather than continuing to
   guess from Telegram-side symptoms.
8. Also found here: `docker compose up -d` doesn't reliably recreate a
   container on a pure `env_file` *content* change if nothing in the
   compose file itself changed — `ops-gateway`'s `StartedAt` was
   unchanged after a supposedly-successful `up -d`. `--force-recreate`
   fixed it. A cousin of the already-documented "restart doesn't
   re-read env_file" gotcha, but catching plain `up -d` too.
9. Wrote `update-notify-bot-token.sh` (same validate+visible-fallback
   pattern) to point `notify.env` at a genuinely different bot
   (`NAS_mancibot`, a fresh one). **Verified for real** — sent two live
   test messages through the ops-gateway's own `/request_approval`
   (using my own svc-claude token, not guessing) and had Christian
   confirm which bot each landed on, rather than trusting `getMe`
   validation alone. Confirmed correctly split: `Hermes_mancibot` for
   chat, `NAS_mancibot` for everything shared/unattended.
