# Session 6 — 2026-07-12 (laptop, Claude Code — Phase 6: ops gateway)

## What was done
Built and fully verified the ops gateway per ADR-011:
- `docker-socket-proxy` (tecnativa/docker-socket-proxy:v0.4.2 — note
  the `v` prefix, the bare `0.4.2` tag doesn't exist despite that being

  four named endpoints: `service_status`, `get_logs`, `disk_usage`,

  `/data/audit.log` regardless of outcome

## Two real bugs hit and fixed during deployment
1. **Wrong image tag**: `tecnativa/docker-socket-proxy:0.4.2` doesn't
   exist — GitHub's releases page showed `v0.4.2` but I dropped the
   `v` prefix when pinning. Docker Hub tags use `v0.4.2`.
2. **`internal: true` network blocks published ports, not just
   outbound routing**: `ops-gateway` was only on the internal network
   shared with `docker-socket-proxy`, so its `ports:` mapping never
   actually bound anything (`docker inspect` showed `"8300/tcp":null`
   despite the compose file listing the mapping). Fix: `ops-gateway`
   needs a SECOND, normal network for the port mapping to work;
   `docker-socket-proxy` stays on the internal-only network alone.
3. (Not a deploy bug, but same session) **Backup marker write failed
   for the Photos job**: added `mkdir -p /volume1/appdata/...` to both
   backup scripts for `backup_health`'s success markers, but
   `restic-photos-backup.sh` runs as `mancinicn`, which has no write
   access to `/volume1/appdata` at all (only root does — same
   asymmetry that made `restic-appdata-backup.sh` need to run as root
   in the first place). Fixed by moving markers to
   `/home/mancinicn/ops-gateway-markers/` — writable by both mancinicn
   directly and root (root can write anywhere).

## Standing access decision (extends ADR-006)
Christian gave Claude the `svc-claude` token for durable, reusable
access — a deliberate step beyond ADR-006's "no standing identity,
borrow the human session" model, made explicitly rather than slipped
in. Storage: local file on the laptop (`C:\Users\jm2_c\.ops-gateway-
token`), outside both git clones (can never be committed), entered via
the same hidden-input pattern as every other secret this project has
used, never pasted through chat. Claude reads it fresh via Bash tool
each call, same as reading `~/.ssh/config` — never re-displays the
value. Revocation: rotate `TOKEN_SVC_CLAUDE` in
`/etc/nas-secrets/ops-gateway.env` and redeploy, independent of
`svc-hermes`.
`TOKEN_SVC_HERMES` — saved to Vaultwarden by Christian, since Hermes
doesn't exist as a running service yet (Phase 10). No config file to
put it into until then.
