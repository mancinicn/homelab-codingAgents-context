## What was done

  (n8n 2.29.8 + postgres:16-alpine, tailnet-bound to
  100.126.31.47:5678), `homelab-infra/nas/home/homeassistant/compose.yml`
  (Home Assistant 2026.7.1, host network, native auth)
- First deploy attempt failed: image extraction error on all
  whiteout-containing layers. Root-caused through two wrong turns to the
  right fix — full diagnosis in **ADR-008**:
  1. UGOS root fs is itself overlayfs → Docker's default data-root
     couldn't do overlay-on-overlay. Moved data-root to
     `/volume1/docker`. Necessary but not sufficient.
  2. Ruled out btrfs/`ugacl` as the cause (`mknod` works fine on
     `/volume1` directly)
  3. Actual cause: Docker 29 defaults to the containerd snapshotter
     (`io.containerd.snapshotter.v1`) for image storage, which failed to
     convert whiteout files on this filesystem stack. Disabled it via
     `daemon.json` (`"features": {"containerd-snapshotter": false}`),
     forcing the classic `overlay2` graphdriver. Both images then pulled
     and extracted cleanly.
- All three containers up: n8n, n8n-postgres (healthy), homeassistant.
  Verified reachable over tailnet from laptop (n8n HTTP 200, HA HTTP 302
  → onboarding redirect, expected)
- Christian confirmed n8n secrets (postgres password, encryption key)
  were saved to Vaultwarden when first generated

  n8n's OIDC SSO is Business/Enterprise-plan-gated (confirmed via
  current n8n docs), Home Assistant has no OIDC support at all. Agreed
  approach instead: native per-app accounts for Christian's wife

  for selective access, plus a shared Vaultwarden org vault for
  password comfort. Real Authentik SSO deferred to Phase 9/10 apps
