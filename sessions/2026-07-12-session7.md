# Session 7 — 2026-07-12 (laptop, Claude Code — Phase 7 v1)

## What was done
Extended the ops gateway (Phase 6) with two write actions, per
ADR-012's deliberately-scoped-down Phase 7 v1:

- **`restart_service`** — via `docker-socket-proxy`'s dedicated
  `ALLOW_RESTARTS=1` flag (checked the proxy's actual env var list
  first rather than assuming; it also has `ALLOW_START`/`ALLOW_STOP`
  as separate granular flags, left off for now — restart is the only
  one actually needed today)
- **`pull_image`** — via `IMAGES=1`+`POST=1`. Confirmed (via the
  proxy's own docs, not assumption) there's no pull-only flag — this
  technically also permits push/remove at the Docker API level.
  Documented as an accepted, low-residual-risk tradeoff in ADR-012
  rather than silently accepted: no registry push credentials exist
  anywhere on this NAS, and in-use image removal is blocked by Docker
  regardless. Pulls an explicit pinned tag per service
  (`SERVICE_IMAGES` mapping in the app) — never `:latest`, consistent
  with the project-wide version-pinning invariant

## Deliberately deferred (see ADR-012 reasoning)
- **`deploy_from_repo`**: a real `compose pull && up -d` needs
  container creation, not just restart — would require broadening the
  proxy to `NETWORKS`+`VOLUMES`+`CONTAINERS`+`POST`, at which point the
  narrow-scope model this gateway is built on stops meaning much. The
  alternative (SSH to `agent_ops`'s existing `docker compose *`
  sudoers rule from inside the gateway) reintroduces the credential
  concern ADR-011 already rejected. Needs its own design pass.
- **Telegram approval flow**: nothing destructive exists yet to gate
  (that's Phase 8) — building the mechanism now would mean testing
  against a synthetic request instead of a real one.
