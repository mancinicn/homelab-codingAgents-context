## Continued same session — Authentik outage + fix

## What was done
- Christian reported `auth.christianmancini.de` unreachable while working
  through the Authentik walkthrough. Diagnosed:
  - DNS resolved correctly; `vault.christianmancini.de` (same VPS, same
    Traefik) responded normally — ruled out DNS/general VPS/network outage
  - `openssl s_client` showed a valid, correctly-scoped Let's Encrypt cert
    for auth.christianmancini.de (not a cert/ACME problem)
  - `curl -v` showed TLS handshake completing and the HTTP request being
    fully sent, then total silence — zero bytes back, full timeout
  - Querying `authentik-server:9000` directly via `docker exec traefik`
    (bypassing Traefik's public router) returned fast, correct responses
    every time, including with the exact external `Host` header — ruled
    out the app itself and Host/ALLOWED_HOSTS validation
  - Traefik's router labels for `authentik` were structurally identical
    to the working `vaultwarden` router (same certresolver, same
    pattern); no conflicting/duplicate routers found across any running
    container; Traefik itself was fully idle (0% CPU, 28MB RAM) — ruled
    out resource exhaustion or a config/label error
  - Conclusion: a stuck keep-alive connection in Traefik's connection
    pool to the `authentik-server` backend — the classic signature when
    every fresh direct connection works but the proxy's reused
    connection hangs indefinitely
- Fix: `docker restart traefik` (confirmed with Christian first — briefly
  interrupts every service behind Traefik, including Vaultwarden).
  Restart took ~3s. Verified after: `auth.christianmancini.de` and
  `vault.christianmancini.de` both HTTP 200; all containers
  (authentik-*, vaultwarden, n8n-zuij) still healthy, unaffected

## Loose ends
- No config change was made — this was a transient stuck-connection
  incident, not a bug in the compose/label setup. If it recurs, the fix
  is the same (restart traefik); if it recurs *often*, worth
  investigating Traefik's keep-alive/idle-timeout settings rather than

## What was done
- Token confirmed working: outpost logs showed "Successfully connected
  websocket" immediately, meaning it was correctly scoped (an
  incorrectly-scoped token would fail here) — the earlier "as user not

- Corrected an earlier mistake in the walkthrough I gave: told Christian
  to use Proxy Provider "Forward auth (single application)" mode, which
  assumes an existing reverse proxy in front of the app. There is none
  on the NAS, so switched to plain "Proxy" mode instead — the outpost
  container itself acts as both auth gate and reverse proxy to n8n, no
  second Traefik needed. Internal host `http://n8n:5678`, external host
  `http://100.126.31.47:5679`
- Multiple false starts before it worked, each diagnosed from outpost
  container logs (`docker logs n8n-outpost` via agent_ops, since
  mancinicn isn't in the docker group either — only agent_ops or sudo
  can read docker):
  1. Provider was never actually saved the first time (silently — the
     "create" wizard was reopened instead of the edit page existing),
     so the outpost logged `"no app for hostname"` for every request.
     Recreated properly this time, confirmed saved (Provider detail
     page showed correct Internal/External host, Mode: Proxy, "Assigned
     to application: n8n (Family)")
  2. Still `"no app for hostname"` after that — turned out the Outpost
     object has its own separate "Applications" field (distinct from
     the Provider's reverse "assigned to application" link) that was
     still empty. Selecting `n8n (Family)` there was the actual missing
     binding.
  3. That triggered a crash loop: `panic: failed to connect to redis:
     dial tcp [::1]:6379: connect: connection refused`. Root cause:
     standalone (non-embedded) proxy outposts need their own Redis for
     session storage — missed this when first writing the compose file.
     Deliberately did NOT point it at the VPS's `authentik-redis` (that
     container has no host port by design, internal-only to the VPS's
     own docker network — reaching across boxes for it would be the
     wrong call). Added a dedicated `redis:7-alpine` container to
     `outpost-compose.yml`, same core-net network, no host port,
     `AUTHENTIK_REDIS__HOST=n8n-outpost-redis` on the outpost
- After the Redis fix: full chain verified working —
  `http://100.126.31.47:5679/` → outpost auth-start → Authentik OAuth2
  authorize endpoint → default authentication flow → **200 OK** on
  Authentik's real login page
