## Continued same session — Authentik outage + fix

## What was done
- Christian reported `auth.christianmancini.de` unreachable while working
  through the Authentik walkthrough. Diagnosed:
  - DNS resolved correctly; `vault.christianmancini.de` (same VPS, same
    Traefik) responded normally — ruled out DNS/general VPS/network outage
  - `openssl s_client` showed a valid, correctly-scoped Let's Encrypt cert
    for auth.christianmancini.de (not a cert/ACME problem)
  - `curl -v` showed TLS handshake completing and the HTTP request being
    fully sent, then total silence — zero bytes back, full timeout
  - Querying `authentik-server:9000` directly via `docker exec traefik`
    (bypassing Traefik's public router) returned fast, correct responses
    every time, including with the exact external `Host` header — ruled
    out the app itself and Host/ALLOWED_HOSTS validation
  - Traefik's router labels for `authentik` were structurally identical
    to the working `vaultwarden` router (same certresolver, same
    pattern); no conflicting/duplicate routers found across any running
    container; Traefik itself was fully idle (0% CPU, 28MB RAM) — ruled
    out resource exhaustion or a config/label error
  - Conclusion: a stuck keep-alive connection in Traefik's connection
    pool to the `authentik-server` backend — the classic signature when
    every fresh direct connection works but the proxy's reused
    connection hangs indefinitely
- Fix: `docker restart traefik` (confirmed with Christian first — briefly
  interrupts every service behind Traefik, including Vaultwarden).
  Restart took ~3s. Verified after: `auth.christianmancini.de` and
  `vault.christianmancini.de` both HTTP 200; all containers
  (authentik-*, vaultwarden, n8n-zuij) still healthy, unaffected

## Loose ends
